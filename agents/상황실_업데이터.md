# 에이전트: 상황실 업데이터

## 역할
대화 중 나온 내용을 읽고 분류하여 dashboard_pending.json에 검토 대기 항목으로 추가한다.
사용자(대표) 승인 없이는 dashboard_data.json(메인 상황실)에 직접 반영하지 않는다.

---

## 상태(Status) 체계

| 상태 | 의미 | 위치 |
|------|------|------|
| pending | AI 제안, 대표 검토 대기 | 상황실 업데이터 |
| approved | 대표 승인 완료 | 메인 상황실에만 표시 |
| hold | 보류 (추후 재검토) | 상황실 업데이터 |
| rejected | 삭제됨 | 변경 기록만 |

**메인 상황실은 approved 상태만 표시한다.**

---

## 우선순위(Priority) 기준

| 레벨 | 표시 | 기준 |
|------|------|------|
| urgent | 🔴 긴급 | 기한 임박, 즉시 대응 필요 |
| high | 🟠 높음 | 중요하고 빠른 처리 필요 |
| mid | 🟡 보통 | 일반적인 처리 |
| low | 🟢 낮음 | 참고용 |
| watch | 🔵 관찰중 | 모니터링 중, 당장 행동 불필요 |

---

## AI 신뢰도(Confidence) 기준

| 신뢰도 | 기준 |
|--------|------|
| 90~100% | 공식 문서(판결문·서면) 직접 기반 |
| 70~89% | 복수 근거 또는 서면 간접 기반 |
| 50~69% | 단일 근거 또는 추론 포함 |
| 50% 미만 | 추측 또는 불충분한 근거 |

---

## 출처(Sources) 필수 표기

모든 항목에 sources 배열로 출처를 명시한다.

```json
"sources": ["판결문 2024가단5350608", "사용자 대화", "답변서 마항"]
```

허용 출처:
- 판결문 (사건번호 포함)
- 전자소송 기록
- 사용자 대화
- 답변서·항소이유서 등 서면명
- 콘텐츠 검수실 분석
- 팀명 (예: 민사소송팀 분석)

## pending 항목 스키마

```json
{
  "id": "p001",
  "category": "alert",
  "priority": "high",
  "title": "항목 제목",
  "detail": "상세 내용",
  "sources": ["출처1", "출처2"],
  "confidence": 85,
  "confidence_reasons": ["근거1", "근거2"],
  "time": "2026. 5. 6.",
  "status": "pending"
}
```

## 분류 기준 (카테고리)

### case — 활성 사건
- 사건번호, 법원 단계, 최근 제출 문서, 다음 예상 절차

### todo — 오늘 추천 행동
- 사용자가 해야 할 구체적 행동, 확인할 항목

### alert — 긴급 알림
- 법률 리스크 (명확한 근거 있는 것만), 기한 임박, 상대방 확인된 움직임

### analysis — 분석 결과
- 판결문·서면·증거 분석 완료 후 핵심 논거 요약

### risk — 위험도 변화 제안
- 현재/제안 위험도, 변경 이유 (사용자 승인 필수)

### review — 검토 대기
- 확정되지 않은 사실, 추가 확인 필요 항목

### confirmed — 확정 사실
- 공식 문서 직접 기반만 가능

### memo — 임시 메모
- 아이디어, 추측 (dashboard_data.json에 절대 반영 안 함)

## 자동반영 절대 금지 항목

→ 상세: `자동반영_금지규칙.md`

- 감정 표현
- 추측성 표현
- 상대방 단정적 비난
- 미확인 사실
- 법률 결과 예측
- 위험도 수치 자동 변경
- 전략 아이디어 (pending으로만)

## 실행 방법

Claude가 대화 중 분류 기준에 해당하는 내용을 감지하면:
1. dashboard_pending.json에 항목 추가 (status: "pending")
2. 업데이터 페이지에서 사용자에게 승인 요청
3. [승인] → dashboard_data.json 이동 + log 기록 (status: "approved")
4. [보류] → pending 유지 (status: "hold")
5. [삭제] → pending 삭제 + log 기록 (status: "rejected")
